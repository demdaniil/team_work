Глоссарий: DDD & Microservices
Эрик Эванс «Domain-Driven Design» · Крис Ричардсон «Microservices Patterns»

1. DDD (Eric Evans)

Стратегический дизайн

  • Domain — предметная область бизнеса 1
  • Subdomain — часть домена (Core / Supporting / Generic) 2
  • Bounded Context — явная граница, внутри которой модель имеет однозначный смысл 3
  • Ubiquitous Language — единый язык команды и кода 4
  • Context Map — карта отношений между контекстами 5

Тактический дизайн

  • Entity — объект с уникальной идентичностью 6
  • Value Object — неизменяемый объект без идентичности 7
  • Aggregate — кластер Entity/VO с единым Aggregate Root 8
  • Repository — абстракция доступа к агрегатам 9
  • Domain Event — факт, случившийся в домене 10
  • Domain Service — логика, не принадлежащая конкретной сущности 11

2. Microservices (Chris Richardson)

Декомпозиция

  • Service — автономный процесс с собственной БД 12
  • Decompose by Subdomain / Business Capability — стратегии нарезки сервисов 13
  • Single Responsibility — один сервис, одна причина меняться 14

Взаимодействие

  • Synchronous vs Asynchronous Messaging — REST/gRPC против брокеров (Kafka, RabbitMQ) 15
  • API Gateway — единая точка входа для клиентов 16
  • Service Discovery — динамический поиск инстансов 17

Данные и согласованность

  • Database per Service — каждый сервис владеет своей схемой 18
  • Saga — распределённая транзакция через цепочку событий/команд 19
  • CQRS — разделение модели чтения и записи 20
  • Event Sourcing — состояние = лог событий 21

Надёжность

  • Circuit Breaker — защита от каскадных отказов 22
  • Idempotency — безопасная повторная обработка сообщений 23
  • Outbox Pattern — атомарная публикация событий вместе с записью в БД 24

Сноски

1 Domain — Предметная область бизнеса. Вся совокупность знаний, правил, процессов и терминологии, которыми оперирует бизнес. Например, для банка домен включает кредитование, счета, платежи, комплаенс. Разработчики и эксперты совместно исследуют домен, чтобы создать точную модель. ↩
2 Subdomain — Часть домена с чёткой ответственностью. Делится на три типа: Core Domain — ключевое конкурентное преимущество компании (сюда вкладываются лучшие ресурсы); Supporting Subdomain — поддерживает Core, но не является уникальным; Generic Subdomain — стандартная функциональность, которую можно купить или взять готовой (например, авторизация, email-рассылка). ↩
3 Bounded Context — Явная граница, внутри которой определённая доменная модель имеет однозначный и согласованный смысл. Один и тот же термин (например, «Клиент») в разных Bounded Context может означать разные вещи и иметь разные атрибуты. Граница фиксируется в коде, схеме данных и командной ответственности. ↩
4 Ubiquitous Language — Единый язык, разделяемый всеми участниками команды (разработчиками, аналитиками, бизнесом) и буквально отражённый в коде: имена классов, методов, переменных. Цель — устранить «потери при переводе» между бизнес-требованиями и реализацией. Язык строго локален для своего Bounded Context. ↩
5 Context Map — Документ или диаграмма, описывающая отношения и интеграции между Bounded Context. Фиксирует паттерны взаимодействия: Shared Kernel (общая часть модели), Customer-Supplier (один контекст зависит от другого), Conformist (подчинение чужой модели), Anti-Corruption Layer (трансляция чужой модели в свою), Open Host Service + Published Language (открытый API с формализованным протоколом). ↩
6 Entity — Доменный объект, обладающий уникальной идентичностью, которая сохраняется на протяжении всего жизненного цикла, даже если его атрибуты меняются. Идентичность обычно выражается через ID. Пример: пользователь, заказ, банковский счёт. Два объекта с одинаковыми атрибутами, но разными ID — разные Entity. ↩
7 Value Object — Объект, не имеющий идентичности; определяется исключительно своими атрибутами. Является неизменяемым (immutable). Два Value Object с одинаковыми атрибутами считаются равными. Примеры: денежная сумма (100 USD), адрес, координаты, диапазон дат. ↩
8 Aggregate — Кластер связанных Entity и Value Object, рассматриваемый как единое целое с точки зрения изменений данных. Каждый Aggregate имеет один корневой объект — Aggregate Root, — через который происходит весь доступ извне. Aggregate определяет границу транзакционной согласованности: все инварианты гарантируются внутри одного Aggregate в рамках одной транзакции. ↩
9 Repository — Абстракция, обеспечивающая доступ к агрегатам так, как если бы они находились в коллекции в памяти. Скрывает детали хранения (SQL, NoSQL, файлы). Позволяет коду домена оставаться независимым от инфраструктуры. Обычно предоставляет методы find, save, delete по идентификатору или критерию. ↩
10 Domain Event — Факт, произошедший в домене и значимый для бизнеса, зафиксированный в прошедшем времени (например, OrderPlaced, PaymentConfirmed). Используется для интеграции между Bounded Context, запуска последующих бизнес-процессов и как основа для Event Sourcing. Является иммутабельным — произошедшее нельзя изменить. ↩
11 Domain Service — Операция или логика предметной области, которая не принадлежит естественным образом ни одной Entity или Value Object. Обычно работает с несколькими агрегатами или выражает сложное бизнес-правило. Примеры: сервис перевода средств между счетами, расчёт маршрута доставки. ↩
12 Service — Автономная, независимо развёртываемая единица функциональности с собственной базой данных и бизнес-ответственностью. Взаимодействует с другими сервисами только через API или сообщения. Позволяет командам разрабатывать, деплоить и масштабировать части системы независимо. ↩
13 Decompose by Subdomain / Business Capability — Два основных подхода к определению границ микросервисов. Decompose by Subdomain опирается на DDD: каждый Bounded Context или Subdomain становится сервисом. Decompose by Business Capability ориентируется на бизнес-функции организации (billing, inventory, shipping). Оба подхода минимизируют зацепление и максимизируют связность внутри сервиса. ↩
14 Single Responsibility Principle (для сервисов) — Каждый сервис должен иметь одну причину для изменения. Применительно к микросервисам: сервис отвечает за одну бизнес-возможность или поддомен, что минимизирует координацию при изменениях и снижает риск непреднамеренных поломок. ↩
15 Synchronous vs Asynchronous Messaging — Два базовых стиля взаимодействия между сервисами. Synchronous (REST, gRPC): вызывающий сервис ждёт ответа; прост в реализации, но создаёт временну́ю связанность и риск каскадных отказов. Asynchronous Messaging (Kafka, RabbitMQ): сервис публикует сообщение и продолжает работу; обеспечивает слабую связанность, высокую отказоустойчивость, но усложняет трассировку и согласованность. ↩
16 API Gateway — Единая точка входа для всех клиентов (мобильных, веб, сторонних). Выполняет маршрутизацию запросов к нужным сервисам, агрегацию ответов, аутентификацию, rate limiting, трансформацию протоколов. Избавляет клиентов от необходимости знать о внутренней топологии сервисов. Паттерн BFF (Backend for Frontend) предполагает отдельный Gateway для каждого типа клиента. ↩
17 Service Discovery — Механизм динамического обнаружения сетевых адресов сервисов. В микросервисной среде инстансы сервисов появляются и исчезают (масштабирование, перезапуск). Client-side Discovery: клиент сам запрашивает реестр (Eureka, Consul) и выбирает инстанс. Server-side Discovery: балансировщик (AWS ALB, Kubernetes Service) делает это прозрачно. ↩
18 Database per Service — Каждый микросервис владеет своей схемой данных и является единственным, кто может к ней обращаться напрямую. Обеспечивает слабую связанность: изменение схемы одного сервиса не затрагивает другие. Усложняет реализацию запросов, охватывающих несколько сервисов (требует API Composition или CQRS). ↩
19 Saga — Паттерн реализации распределённых транзакций без использования двухфазного коммита (2PC). Длинная транзакция разбивается на последовательность локальных транзакций, каждая из которых публикует событие или команду для следующего шага. При отказе выполняются компенсирующие транзакции. Choreography Saga: сервисы реагируют на события друг друга. Orchestration Saga: центральный оркестратор управляет последовательностью шагов. ↩
20 CQRS (Command Query Responsibility Segregation) — Паттерн разделения модели записи (Command) и модели чтения (Query). Команды изменяют состояние через Aggregate; для чтения создаются отдельные денормализованные проекции, оптимизированные под конкретные запросы. Особенно полезен в микросервисах для реализации запросов, охватывающих несколько сервисов. ↩
21 Event Sourcing — Паттерн хранения состояния: вместо текущего состояния объекта хранится полная хронологическая последовательность Domain Event. Текущее состояние восстанавливается путём воспроизведения (replay) событий. Даёт полный аудит-лог, упрощает отладку и естественно интегрируется с CQRS. Усложняет эволюцию схемы событий и требует механизмов снапшотов при большом числе событий. ↩
22 Circuit Breaker — Паттерн защиты от каскадных отказов. Обёртка вокруг вызова удалённого сервиса, которая отслеживает количество ошибок. При превышении порога «размыкает цепь»: последующие вызовы сразу возвращают ошибку (или fallback), не дожидаясь таймаута. После паузы пробует восстановить соединение. Реализации: Resilience4j, Hystrix. ↩
23 Idempotency — Свойство операции: её многократное выполнение с одинаковыми входными данными даёт тот же результат, что и однократное. Критично в распределённых системах, где сообщения могут быть доставлены более одного раза (at-least-once delivery). Реализуется через уникальный идентификатор операции (idempotency key) и проверку уже обработанных запросов. ↩
24 Outbox Pattern — Паттерн гарантированной публикации событий. Вместо прямой отправки события в брокер, сервис атомарно записывает и бизнес-данные, и событие в локальную таблицу Outbox в рамках одной транзакции с БД. Отдельный процесс (Message Relay) читает Outbox и публикует события в брокер. Исключает ситуацию, когда данные сохранены, а событие не отправлено (или наоборот). ↩
